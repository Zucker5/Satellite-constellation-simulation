### 1.1 Выбор концепции
В процессе реализации данного этапа было принято решение отойти от общепринятой концепции ООП в сторону ECS. Ключевое отличие заключается в подходе к созданию объектов. Классический ООП подход предполагает инкапсуляцию данных и логики внутри классов. В контексте моделирования множества спутников это создает проблему, каждый объект является «тяжелым» контейнером. При работе с тысячами объектов это приводит к фрагментации памяти и низкой эффективности кэша процессора, так как объекты разбросаны по «куче», а их методы могут скрывать избыточные обращения к памяти. ECS разделяет данные (Компоненты) и логику (Системы). Компоненты — это простые структуры, не содержащие методов. Благодаря этому данные одного типа располагаются в памяти последовательно (Contiguous memory). Когда процессор обрабатывает массив таких компонентов, он максимально эффективно использует кэш-линии, что повышает производительность

Рассмотрим непосредственно на примере кода. Допустим, нам необходимо создать три объекта в нашем виртуальном мире, пусть это будет ```Player```, ```NPC``` и ```Home```. Очевидно, что ```Player```и ```NPC``` обладают и координатами, и вектором движение, а дом только координатами. Исходя из этого, по принципу ООП, нам необходимо создать три класса для каждого объекта.
```
class home{

public:
        
private:
        int x; // координаты
        int y;
};
```

```cpp
class Player{

public:
        // реализация методов для доступа к переменным класса
private:
        int x; // координаты
        int y;

        int dx; // вектор движения
        int dy;
};
```

```
class  NPС{

public:
        // реализация методов для доступа к переменным класса
private:
        int x; // координаты
        int y;

        int dx; // вектор движения
        int dy;
};
```

Как видно из примера, в каждом классе отдельно создаются методы, для взаимодействия со свойствами объекта.

Теперь, при тех же условия, реализация по концепции ECS.
```
struct Position
{
        int x;
        int y;
};
```

```
struct Vector
{
        int dx;
        int dy;
};
```

```
Position f_position() {
        // реализация функции для взаимодействия с координатами
};
```

```
Vector f_vector() {
        // релизация функции для домтупа к вектору движения
};
```

```
entt::registry registry; // используя специальную библиотеку entt для ECS создаем регистр, в котором хранятся наши объекты

auto Player = registry.create(); // создаем объект Pleyer
auto Home = registry.create(); // создаем объект Home
auto NPС = registry.create();// создаем объект NPС
        

registry.emplace<Position>(Player); // присваиваем объекту Player свойства Position
registry.emplace<Vector>(Player); // присваиваем объекту Player свойства Vector

registry.emplace<Position>(NPС); // присваиваем объекту NPС свойства Position
registry.emplace<Vector>(NPС);// присваиваем объекту NPС свойства Vector


registry.emplace<Position>(Home); // присваиваем объекту Home свойства Position
```

Обратим внимание, что, при использовании концепции ECS, мы только единожды создаем методы для взаимодействия с свойствами объекта, а потом взаимодействуем ими с любым объектом из регистра.

В случае ООП, если вам нужно обновить только координаты (x, y) у 1000 NPC, чтобы вычислить их новое положение, процессор вынужден загрузить в кэш весь объект. А внутри объекта могут лежать еще и параметры здоровья, инвентаря, текстур, звуковых эффектов и так далее. А в случае ECS в кэш загрузится только необходимый Position, что позволяет более эффективно взаимодействовать с памятью

1.2 Логика и реализация создания спутников

Поскольку цель нашей симуляции не доскональный расчет физики полета спутника на орбите земли, а реализовать алгоритмы для поиска оптимальных маршрутов передачи данных между двумя точками на Земле, посредством спутниковой связи, решено использовать готовую аналитическую модель sgp4, которая используется для отслеживание спутников, визуализация орбит, расчёт сближений объектов, планирование миссий. Эта модель предсказывает влияние возмущений, вызванных формой Земли, сопротивлением атмосферы, радиацией и гравитационным воздействием других тел, таких как Солнце и Луна, чего достаточно для наших целей. На вход sgp4 подается TLE.

TLE — двухстрочный формат данных, представляющий собой набор элементов орбиты для спутника Земли. Этот формат имеет строго установленную структуру и количество символов в строке

1.3 Логика обновления координат спутниками

После того, как у нас есть спутники, за которыми закреплены координаты и TLE, задача обновления координат с течением времени становится тривиальной.

С запуском программы, мы запускаем таймер, который и будет симулировать течение реального времени. Этот таймер идет от запуска, до завершения программы.

Модель  sgp4 высчитывает положение на орбите в  конкретный момент времени, а в TLE у нас содержится дата введения спутника в строй, все что нам нужно, это сложить таймер и дату вывода спутника, так мы получит конкретный момент времени. Так же, необходимо учесть, что sgp4 использует юлианскиую дату, то есть необходимо реализовать перевод текущего момента времени в юлианскую дату.
```

//функция, которая переводит герегорианскую дату в юлианскую
```

```

```
